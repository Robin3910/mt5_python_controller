## 1. 背景与目标

参考库 `mt5_python_connector` 是一个**单机版** Flask 应用：在一台装有 MT5 的 Windows 机器上接收 TradingView Webhook 信号，并直接在本机 MT5 上下单。

本方案要将其扩展为**一拖 N 的多节点跟单/信号分发系统**：

- **一个中心管理后台**（FastAPI + Redis）：统一接收信号、管理所有 MT5 节点、按规则分发信号、实时监控所有账户、提供 Web 管理界面。
- **N 个 MT5 节点客户端**：每个节点运行在一台（云）电脑上，本地连接自己的 MT5 终端，通过 WebSocket 长连接接入后台，负责上报状态、接收并执行交易指令。
- **Web 管理端**：PC 与手机 H5 自适应，用于配置、监控、远程平仓。

> **关键约束（需求）**
>
> 1. 节点接入需要鉴权——采用**所有节点共享一个全局 `NODE_TOKEN`**（管理员可在「配置」页查看/重置），节点身份由 `MT5_LOGIN` 唯一识别；客户端用同一 `MT5_LOGIN` 首次接入时若库中尚未注册会按默认配置自动入库（也支持管理员手动预先创建）。
> 2. **信号接收规则、解析逻辑、MT5 操作规则**与参考库代码保持一致（本方案在第 8、10 章逐条对齐）。

### 1.1 与参考库的职责拆分

参考库的三块核心逻辑被拆分到分布式架构的不同位置，但**逻辑本身保持不变**：


| 参考库模块                   | 原职责               | 新架构中的位置                             |
| ----------------------- | ----------------- | ----------------------------------- |
| `tradingview_parser.py` | 解析 Webhook 信号     | **管理后台**（信号入口处解析一次）                 |
| `app.py` 的持仓判断逻辑        | 开仓前检查是否已有持仓       | **管理后台分发引擎**（基于全局实时持仓快照决策） + 节点本地兜底 |
| `mt5_client.py`         | 连接/下单/平仓/查询       | **MT5 节点客户端**（在本地 MT5 执行）           |
| `config.py`             | 符号映射 / 关键字 / 默认参数 | 后台与节点共享的公共配置模块                      |

### 2.1 数据流（开仓信号为例）

1. TradingView 触发 Alert → `POST /webhook` 到达管理后台。
2. 后台校验来源（Token / IP 白名单），用 `TradingViewParser` 解析为标准 `TradingSignal`。
3. 分发引擎读取 Redis 中各节点的**实时持仓快照**与**配置**，按规则筛选目标节点：
  - 多区间方向过滤（表 4）
  - 有持仓禁止再开仓 / 无持仓才可接收（表 5、6）
  - 计算每个节点的下单手数（统一 / 独立，表 3）
  - 根据当前模式分发：**全员同步**（广播）或 **轮询领取**（队列，表 7、8）
4. 通过 WebSocket 把交易指令下发到目标节点。
5. 节点用 `MT5Client`（逻辑同参考库）在本地 MT5 执行，回传执行结果。
6. 后台记录结果、刷新账户/持仓快照，并通过 WebSocket 推送给 Web 管理端实时展示。

## 3. 技术选型


| 层         | 选型                                                                               | 说明                                                           |
| --------- | -------------------------------------------------------------------------------- | ------------------------------------------------------------ |
| Web 框架    | **FastAPI** + Uvicorn                                                            | 原生异步、内置 WebSocket、自动 OpenAPI 文档                              |
| 异步运行时     | asyncio                                                                          | 单进程内管理大量 WS 长连接                                              |
| 缓存/状态/消息  | **Redis 7**                                                                      | 节点状态、账户快照、信号队列、Pub/Sub、分布式锁                                  |
| Redis 客户端 | `redis-py`（asyncio 模式）                                                           | 异步读写与 Pub/Sub                                                |
| 节点 ↔ 后台   | **WebSocket (wss)**                                                              | 长连接、双向、低延迟                                                   |
| MT5 接入    | `MetaTrader5`（官方 Python 包）                                                       | 仅运行在节点侧（Windows）                                             |
| 数据校验      | Pydantic v2                                                                      | 请求/消息模型校验                                                    |
| 持久化       | **MySQL 8**（SQLAlchemy 2.0 + asyncmy 异步驱动；Alembic 迁移）                            | 节点账号、操作审计、历史信号（Redis 仅存实时态）                                  |
| 鉴权        | JWT（管理员）/ Token（节点）/ Bearer+IP（Webhook）                                          | 见第 14 章                                                      |

| 部署        | 后台 Linux/Windows 均可；节点必须 Windows                                                 | 节点依赖 MT5 终端                                                  |

## 4. 核心概念与数据模型

### 4.1 节点（Node）

代表一台运行 MT5 的客户端机器（一个 MT5 账户）。**`mt5_login` 是节点的业务唯一键**——既是接入鉴权的识别依据，也是 Web 列表 / 持仓快照的归并 key；管理端不再保存每节点 token（鉴权改为全局共享，见 §6 与 §4.5）。

```python
class Node(BaseModel):
    node_id: str            # 系统生成的唯一 ID（如 nd_xxxxxxxx）
    name: str               # 人工备注名（留空时自动生成 "node-<mt5_login>"）
    enabled: bool = True    # 是否启用（禁用后不分发、不允许连接）
    status: str = "offline" # online / offline
    # —— 按币种分发配置（filters 字典，键为品种代码）——
    # 每项可含：follow_sync, follow_poll, lot_mode, lot, poll_order
    filters: dict | None = None
    # —— MT5 绑定（自 v0.2 起，mt5_login 创建后不可修改，且全局唯一）——
    mt5_login: int                # 必填，> 0
    mt5_server: str | None = None # 由节点 hello 消息上报后回填
    created_at: float
    last_seen: float | None = None
```

> **自动注册默认值**：`name="node-<login>"`、`enabled=False`（须管理员启用后才可接入）；`filters` 从中控台已登记品种**自动生成**（每个品种一条：`follow_sync=True`、`follow_poll=True`、`lot_mode=fixed`、`lot=0.01`、`poll_order=0`）。见 §6 与 `node_service.default_node_filters_from_global`。
>
> **v0.4 变更**：节点**必须**在 `filters` 中为信号品种配置按币种条目，否则该节点拒收（`skip_reason`：`节点未配置：{sym}未在节点按币种配置中，拒收`）；不再对未配置品种回退节点级默认策略。
>
> **v0.3 变更**：原节点级 `lot_mode` / `follow_sync` / `follow_poll` / `poll_order` 已下沉到 `filters` 的**按币种条目**；库表仍保留旧列作兼容，运行期以 `filters` 为准。

### 4.2 账户快照（AccountSnapshot）

节点周期性上报，存于 Redis，供监控与分发决策使用。

```python
class Position(BaseModel):
```

### 4.3 交易信号（TradingSignal）

**与参考库 `tradingview_parser.py` 的 `TradingSignal` 完全一致**，仅新增系统调度字段：

```python
class TradingSignal(BaseModel):
    action: str                 # BUY / SELL / CLOSE
    symbol: str                 # 已映射为 MT5 符号
    volume: float
    stop_loss: float | None = None
    take_profit: float | None = None
    order_type: str | None = "market"
    comment: str = ""
    allow_position: bool = False   # 与库一致：为 True 时允许已有持仓仍开仓
    # —— 系统新增（调度用，不影响 MT5 下单语义）——
    signal_id: str                 # 后台生成，幂等/追踪/容错用
    received_at: float
```

### 4.4 交易指令（Command，后台 → 节点）

```python
class Command(BaseModel):
    cmd: str                     # open / close / config / query / ping
    signal_id: str | None = None
    # open 时：
    action: str | None = None    # BUY / SELL
    symbol: str | None = None
    volume: float | None = None
    stop_loss: float | None = None
    take_profit: float | None = None
    comment: str = ""
    magic: int | None = None
    # close 时：
    close_target: str | None = None   # all / symbol / ticket
    close_symbol: str | None = None
    close_ticket: int | None = None
```

### 4.5 系统级配置（SystemSetting，v0.2 新增）

以 key/value 形式持久化系统级配置。当前唯一用途：保存全局节点接入令牌 `node_token`。

```python
class SystemSetting(BaseModel):
    key: str            # e.g. "node_token"
    value: str          # 明文存储（节点令牌需要复制到每个 .env，因此不哈希）
    updated_at: float
```

> 设计权衡：与单节点 token 不同，**全局令牌需要管理员能随时查看以复制到节点 `.env`**（同一令牌可能分发到 N 台节点），因此采用「明文持久化 + Redis 缓存 + 管理端可显示/重置」策略，而非"仅哈希、只在生成时显示一次"。读取与重置端点都受管理员 JWT 鉴权保护（见 §13.3）。

## 5. 数据存储设计

> **Redis** 负责**实时态**（节点在线状态、账户快照、信号队列、Pub/Sub、分布式锁）；**MySQL** 负责**持久化权威源**（节点注册与配置、操作审计、历史信号与逐节点执行结果）。Redis 中的节点元数据视为 MySQL 的热缓存，服务启动时从 MySQL 预热。

### 5.1 Redis 键设计（实时态）


| Key                         | 类型              | 内容                                                              | TTL             |
| --------------------------- | --------------- | --------------------------------------------------------------- | --------------- |
| `node:{node_id}`            | String(JSON)    | 节点元数据 + 交易配置（见 Node 模型）                                         | 持久              |
| `node:by_login:{mt5_login}` | String          | → `node_id`（WS 握手时按 `mt5_login` 反查节点；v0.2 起替代了原 `node:token`） | 持久              |
| `nodes`                     | Set             | 全部 `node_id`                                                    | 持久              |
| `node:online:{node_id}`     | String          | `"1"`，心跳续期，作为"在线"判据                                             | 心跳间隔 × 3（如 90s） |
| `node:account:{node_id}`    | String(JSON)    | 最新 `AccountSnapshot`                                            | 持久（被覆盖）         |
| `signal:{signal_id}`        | Hash            | 信号原文 + 状态（pending/dispatching/done/failed）                      | 24h             |
| `signal:poll:pending`       | List            | 轮询模式：待处理信号队列（信号之间串行排队）                                          | 持久              |
| `signal:poll:{signal_id}`   | String(JSON)    | 轮询信号处理结果：信号快照 + 领取状态 `status`(pending/done/unconsumed) + 领取节点 `consumer` | 24h             |
| `signal:poll:rotation:{symbol}` | String(JSON) | **按品种的轮转顺序**：`node_id` 有序列表；每次被领取的节点移到队尾，实现循环轮转（round-robin） | 持久              |
| `config:lot:global`         | String          | 全局统一手数配置（启用/数值）                                                 | 持久              |
| `config:filters`            | String(JSON)    | **按品种全局规则**（准入清单 + 方向总开关 + 分发模式/持仓范围 + 区间过滤；见 9.2） | 持久              |
| `config:node_token`         | String          | 全局节点接入令牌（明文缓存；权威源在 MySQL `system_setting` 表，见 §4.5 与 §5.2）  | 持久              |
| `cmd:{node_id}`             | Pub/Sub Channel | 多实例部署时，向某节点路由指令                                                 | —               |
| `lock:dispatch:{signal_id}` | String(NX/EX)   | 分发幂等锁，防重复处理                                                     | 30s             |


> **单实例 vs 多实例**：单实例部署时，WebSocket 连接对象可直接保存在进程内字典 `connections[node_id]`，分发即直接 `await ws.send_json(...)`。多实例水平扩展时，每个实例订阅自己持有连接的 `cmd:{node_id}` 频道，分发方 `PUBLISH` 指令，由持有连接的实例转发——以此解耦"分发逻辑"与"连接归属"。

### 5.2 MySQL 表结构（持久化）

> 引擎 InnoDB，字符集 `utf8mb4`；事件类表（审计 / 历史 / 执行）默认保留 **90 天**（定时清理或按天分区）。SQLAlchemy 2.0 + asyncmy，迁移用 Alembic。

```sql
-- 节点注册与配置（权威源；Redis 为热缓存）
-- v0.2 起：移除 token_hash；mt5_login 升级为 NOT NULL + UNIQUE，作为节点的业务唯一键
CREATE TABLE nodes (
    node_id      VARCHAR(32)   PRIMARY KEY,
    name         VARCHAR(64)   NOT NULL,
    enabled      TINYINT(1)    NOT NULL DEFAULT 1,
    lot_mode     VARCHAR(16)   NOT NULL DEFAULT 'global',-- global / fixed / signal
    lot          DECIMAL(10,2) NULL,
    follow_sync  TINYINT(1)    NOT NULL DEFAULT 1,
    follow_poll  TINYINT(1)    NOT NULL DEFAULT 1,
    poll_order   INT           NOT NULL DEFAULT 0,       -- 轮询消费顺序，默认按 created_at
    filters_json JSON          NULL,                     -- 节点级区间过滤(未配置回落全局)
    mt5_login    BIGINT        NOT NULL,
    mt5_server   VARCHAR(64)   NULL,
    created_at   DATETIME(3)   NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    updated_at   DATETIME(3)   NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
    UNIQUE KEY uk_mt5_login (mt5_login)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 系统级配置（v0.2 新增）：当前唯一用途是保存全局节点接入令牌 node_token
CREATE TABLE system_setting (
    `key`       VARCHAR(64)  PRIMARY KEY,
    `value`     TEXT         NOT NULL,
    updated_at  DATETIME(3)  NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 操作审计（管理端敏感操作）
CREATE TABLE audit_log (
    id          BIGINT       PRIMARY KEY AUTO_INCREMENT,
    ts          DATETIME(3)  NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    operator    VARCHAR(64)  NOT NULL,
    action      VARCHAR(64)  NOT NULL,                   -- create_node/rotate_node_token/close_all/update_config...
    target      VARCHAR(64)  NULL,                       -- node_id / 全局
    params_json JSON         NULL,
    result      VARCHAR(16)  NOT NULL,                   -- ok / fail
    ip          VARCHAR(45)  NULL,
    KEY idx_ts (ts), KEY idx_operator (operator)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 历史信号（webhook 接收与解析结果）
CREATE TABLE signal_history (
    signal_id     CHAR(32)      PRIMARY KEY,
    received_at   DATETIME(3)   NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    source_ip     VARCHAR(45)   NULL,
    raw_payload   TEXT          NULL,                    -- 原始 webhook 内容
    action        VARCHAR(8)    NULL,                    -- BUY / SELL / CLOSE
    symbol        VARCHAR(32)   NULL,
    volume        DECIMAL(10,2) NULL,
    sl            DECIMAL(18,5) NULL,
    tp            DECIMAL(18,5) NULL,
    comment       VARCHAR(128)  NULL,
    parsed_ok     TINYINT(1)    NOT NULL DEFAULT 0,
    dispatch_mode VARCHAR(8)    NULL,                    -- sync / poll
    status        VARCHAR(16)   NOT NULL DEFAULT 'pending', -- pending/dispatching/done/partial/failed
    KEY idx_recv (received_at), KEY idx_symbol (symbol)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 逐节点执行结果（一条信号 × 一个节点 = 一行；同步/轮询通用，用于对账）
CREATE TABLE signal_dispatch (
    id            BIGINT        PRIMARY KEY AUTO_INCREMENT,
    signal_id     CHAR(32)      NOT NULL,
    node_id       VARCHAR(32)   NOT NULL,
    decided_vol   DECIMAL(10,2) NULL,                    -- 实际下单手数(9.4)
    gate_result   VARCHAR(16)   NOT NULL,                -- passed / skipped
    skip_reason   VARCHAR(64)   NULL,                    -- has_position/filtered/offline...
    status        VARCHAR(16)   NOT NULL DEFAULT 'pending', -- pending/done/failed/skipped
    retcode       INT           NULL,                    -- MT5 retcode(10.7)
    order_ticket  BIGINT        NULL,
    deal          BIGINT        NULL,
    error         VARCHAR(255)  NULL,
    dispatched_at DATETIME(3)   NULL,
    finished_at   DATETIME(3)   NULL,
    KEY idx_signal (signal_id), KEY idx_node (node_id), KEY idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

> 账户实时态（balance/equity/持仓）存于 Redis；如需历史曲线/留痕对账，可另建 `account_snapshot` 时序表（数据量大，建议按天分区或转时序库，按需开启）。

## 6. 节点接入鉴权与自动注册

> **核心规则（v0.2）：所有节点共享一个全局 `NODE_TOKEN`（在「配置」页查看/重置），节点身份由 `MT5_LOGIN` 唯一标识；首次接入的 `MT5_LOGIN` 自动按默认配置入库。** Token 不正确、缺少 `MT5_LOGIN`、节点被禁用或重复登录的连接均被拒绝。

### 6.1 与 v1.0（每节点一 token）的差异

| 项 | v1.0（已废弃） | v0.2（当前） |
| --- | --- | --- |
| 鉴权令牌 | 每节点一份，明文仅创建时显示一次，DB 存 sha256 | 全局共享一份，明文持久化，可在「账户设置」随时查看/重置 |
| 节点身份识别 | token → `node_id` 反查 | 校验全局 token → 用 `mt5_login` 反查/自动创建 |
| 节点入库 | 仅管理员手动 `POST /api/nodes` | 手动 `POST /api/nodes`（可选）+ node_client 首次登录自动注册 |
| `mt5_login` 约束 | 可空、可重复 | NOT NULL、全局 UNIQUE |
| API 端点 | `POST /api/nodes/{id}/rotate-token` | `GET/POST /api/config/node-token[/rotate]` |
| 节点 `.env` | 各自不同的 token | 同一个全局 token + 各自的 `MT5_LOGIN` |

### 6.2 全局令牌生命周期

```
首次启动后端
   │
   ▼
ensure_node_token(): SELECT system_setting WHERE key='node_token'
   │  存在  → 把 value 同步到 Redis 缓存
   │  缺失  → secrets.token_urlsafe(32) 写库 + 缓存
   ▼
管理员 GET  /api/config/node-token             → 返回 {token, updated_at}（管理员 JWT 鉴权）
管理员 POST /api/config/node-token/rotate      → 重新生成 + 写库 + 刷缓存 + 落审计
                                              ↳ 所有节点的旧 .env 立即失效，需要同步更新
```

要点：
- **存明文**：节点 `.env` 需要这份 token，管理员必须能多次复制；同时端点受 JWT 保护，未登录无法读取。
- **常量时间比较**：握手用 `hmac.compare_digest(client_token, expected_token)`，与参考库 `verify_auth` 同源的安全实践。
- **DB 权威 + Redis 缓存**：DB 是权威源，启动 / 重置时刷新缓存；缓存失效时回源 DB。
- **审计**：`rotate_node_token` 记录到 `audit_log`（操作人 / IP / 时间）。

### 6.3 节点身份识别与自动注册

```
node_client 启动
   │ 读 .env: NODE_TOKEN, MT5_LOGIN, MT5_PASSWORD...
   ▼
建立 wss://hub/ws/node 连接 → ws.accept()
   │
   ▼ 首包
   ws.send {"type":"auth","data":{"token":"<NODE_TOKEN>","mt5_login":<int>}}
   │
   ▼ 后台校验顺序
   1. 拉取全局 token（Redis → DB 回源）；hmac.compare_digest 比较
        ✗  → auth_fail(invalid_token) + close 4401
   2. 解析 mt5_login（必须为正整数）
        ✗  → auth_fail(missing_mt5_login) + close 4401
   3. 反查节点：Redis `node:by_login:{login}` → 命中即取节点；未命中再 DB 直查兜底
   4. 仍不存在 → node_service.auto_register(login)
        ├─ 默认配置：name=f"node-{login}", enabled=False（入库后本轮鉴权仍会因禁用被拒 4403）
        ├─ filters：从中控台已登记品种批量生成（每品种 follow_sync/poll=True, lot_mode=fixed, lot=0.01, poll_order=0）
        └─ 写库 + 刷缓存 + broadcast_admin({type:"node_registered",...})
   5. 节点 enabled=False  → auth_fail(disabled) + close 4403
   6. 同 node_id 已在线   → auth_fail(already_online) + close 4409
   7. 通过 → register_connection + send auth_ok({node_id, heartbeat})
```

要点：
- **token 不放 URL**：query string 会进入 Nginx/反代访问日志，是敏感信息泄露面。统一用**连接后首包 `auth`**。
- **首包超时**（`AUTH_FIRST_PACKET_TIMEOUT` 默认 5s）→ 关闭码 `4400`，防止未鉴权连接占用资源。
- **关闭码约定**：`4400` 握手超时/格式错误 / `4401` 鉴权失败（invalid_token / missing_mt5_login）/ `4403` 节点被禁用 / `4409` 重复连接 / `4500` 自动注册失败（DB 异常）。
- **重复连接**：同一 `node_id` 已有存活连接时拒绝新连接（`already_online`）；旧连接断开后新连接才能接入——避免互相踢线导致连接雪崩。

参考实现：`backend/app/ws_gateway.py::node_ws` + `backend/app/node_service.py::auto_register` + `backend/app/system_settings.py::get_node_token`。

## 7. WebSocket 长连接与节点管理（表 1 / 表 2）

> 对应表格第 1 条（心跳检测、信息上报、长连接建立、状态管理、传输通道）与第 2 条（实时监控所有账户、上报与解析）。

### 7.1 消息协议

统一 JSON 信封：`{ "type": "...", "data": {...}, "ts": 1719... }`

**节点 → 后台**


| type           | 说明               | data 关键字段                              |
| -------------- | ---------------- | -------------------------------------- |
| `auth`         | （可选）握手后再次声明身份/版本 | client_version                         |
| `heartbeat`    | 心跳               | ts                                     |
| `account`      | 账户 + 持仓快照上报      | AccountSnapshot                        |
| `trade_result` | 指令执行结果           | signal_id, success, order, deal, error |
| `ack`          | 轮询模式领取/完成确认      | signal_id, stage(claimed/done)         |


**后台 → 节点**


| type                    | 说明                 | data 关键字段                 |
| ----------------------- | ------------------ | ------------------------- |
| `auth_ok` / `auth_fail` | 鉴权结果               | node_id                   |
| `command`               | 交易/平仓/查询指令         | Command 模型                |
| `config`                | 下发交易配置（手数/区间/模式开关） | lot_mode, lot, filters... |
| `pong`                  | 心跳回应               | ts                        |


### 7.2 心跳与状态管理

- 节点每 `HEARTBEAT_INTERVAL`（默认 30s）发送 `heartbeat`；后台收到后 `SET node:online:{id} 1 EX 90`（3 倍间隔容错）。
- 后台同时回 `pong`，节点据此判断后台可达性；连续丢失则触发重连（指数退避）。
- "在线"判据：`node:online:{id}` 存在即在线；TTL 到期自动判定离线。
- 后台维护一个 watchdog 协程，定期扫描在线集合变更，将 `online/offline` 变化通过 WS 推送给 Web 管理端。
- 连接断开（`WebSocketDisconnect`）：立即清理进程内连接表，并删除/缩短 `node:online` 标记。

### 7.3 账户实时监控（表 2）

- 节点每 `ACCOUNT_REPORT_INTERVAL`（默认 5s）调用本地 MT5 获取 `account_info()` + `positions_get()`，组装 `AccountSnapshot` 上报。
- 后台解析后写入 `node:account:{id}`，并增量推送到 Web 管理端。
- 该快照是**分发引擎做持仓过滤的依据**（表 5、6），因此上报频率要兼顾实时性与负载；下单/平仓等关键动作后，节点应**立即触发一次额外上报**以减小决策延迟窗口。

### 7.4 重连与可靠性

- 节点侧：断线后自动重连（退避 1s→2s→…→30s 上限），重连成功后立即补发一次账户快照。
- 后台侧：指令下发使用 `signal_id` 幂等；节点对重复 `signal_id` 去重，避免重连导致重复下单。

## 9. 信号分发引擎（表 3~8）

分发引擎是新增的核心模块，输入一条 `TradingSignal`，输出"对哪些节点、用什么手数、用什么模式下发什么指令"。

### 9.1 总体决策流程

```
            ┌─────────────────────────────────────────────┐
   signal ─▶            │ 0) 品种准入：signal.symbol 须在中控台 filters 中登记且 enabled=true │
            │     未登记 / 已禁用 → 拒收（webhook status=rejected；含 CLOSE） │
            │ 1) 候选节点 = 在线 ∧ enabled ∧ 节点 filters 已配该品种 │
            │     ∧ 该币种参与对应模式（未配品种 → skipped 并记日志）│
            │ 2) 方向总开关 + 多区间方向过滤（表4）          │
            │ 3) 持仓过滤（表5/6，scope 按品种配置）          │
            │ 4) 计算每节点手数（表3，按节点 filters 币种项）  │
            │ 5) 按该品种 dispatch_mode 分发：               │
            │     - sync(表7): 过滤后节点并发同时执行         │
            │     - poll(表8): 按轮转顺序选 1 个节点领取     │
            └─────────────────────────────────────────────┘
```

**分发模式与持仓范围**（v0.3）：不再使用全局 `config:dispatch` API，改为写入 `config:filters` 的**每个品种条目**（`dispatch_mode`、`position_scope`）。环境变量 `DISPATCH_MODE` / `POSITION_SCOPE` 仅作该字段缺省时的回退默认。

### 9.2 多区间方向过滤（表 4）

**目的**：在不同价格区间内只允许特定方向开仓（例如：金价 2300–2350 只允许做空，2250–2300 只允许做多），用于风控与策略过滤。

配置结构（`config:filters`，**按品种为键**；未登记或 `enabled=false` 的品种 webhook 直接拒收，含 CLOSE；后台手动平仓不受影响）：

```json
{
  "XAUUSD": {
    "enabled": true,
    "allow_buy": true,
    "allow_sell": true,
    "dispatch_mode": "sync",
    "position_scope": "symbol",
    "default_action": "block",
    "intervals": [
      { "low": 2300, "high": 2350, "allow": ["SELL"] },
      { "low": 2250, "high": 2300, "allow": ["BUY"] }
    ]
  }
}
```

判定逻辑（开仓信号才参与区间/持仓过滤；CLOSE 不受区间/持仓过滤，但仍须品种已登记且 `enabled=true`）：

0. **品种启用开关**：`enabled=false` 时拒收该品种全部信号（含 CLOSE）。
1. **方向总开关**（v0.3）：`allow_buy` / `allow_sell` 为 `false` 时，直接拦截对应方向。
2. 取该符号当前价格——**以该节点实时价格为准**（后台下发前向该节点查询当前 tick，或将区间规则随指令下发、由节点执行前本地校验），避免用陈旧快照误判。
3. 找到价格落入的区间；若该区间 `allow` 不含本次方向 → **拦截**。
4. 不落入任何区间时按 `default_action` 处理。

> **配置层级（已确认）**：
> - **中控台（全局）**：`config:filters` 登记品种并配置分发模式、持仓范围、方向总开关与区间规则；同时作为 Webhook **准入清单**。
> - **节点级**：`node.filters` 按币种配置 `follow_sync` / `follow_poll` / `lot_mode` / `lot` / `poll_order`；**未单独配置的币种对该节点拒收**（v0.4；自动注册时从中控台已有品种批量生成默认条目）。
> - 区间过滤在节点侧与全局**字段级合并**（`effective_filters`）：节点可覆盖全局同名品种的 `enabled`、区间等字段。

### 9.3 持仓过滤：有持仓禁止再开仓 / 无持仓才接收（表 5、6）

**与参考库 `app.py::execute_trade` 的语义对齐**：库中在 BUY/SELL 前，若 `allow_position=False` 且该 **symbol 已有持仓**，则跳过。

本系统把该判断**前移到后台**（因为后台掌握所有节点的实时持仓快照），实现"无持仓账户才下发、有持仓账户不下发"：

```python
def position_gate(node_id, signal, snapshot, scope="symbol"):
    if signal.action not in ("BUY", "SELL"):
        return True               # 平仓不受此限
    if signal.allow_position:
        return True               # 与库一致：显式允许则放行
    held = snapshot.positions
    if scope == "symbol":         # 默认：与库一致，按 symbol 判断
        held = [p for p in held if p.symbol == resolve(signal.symbol)]
    # scope == "account": 账户级——只要有任意持仓就拦截
    return len(held) == 0
```

- `**scope="symbol"`（默认）**：完全对齐参考库（同一品种有持仓才拦截）。
- `**scope="account"`**：对齐表格字面"有持仓账户禁止再次开仓"（只要账户有任意持仓即拦截）。
- **（已确认）两种范围都实现**，由 `config:dispatch.position_scope` 在管理端运行时切换，**默认 `symbol`**（对齐参考库）。
- **节点侧兜底**：节点执行开仓前再次本地校验持仓（库原逻辑保留），双保险防止快照延迟导致的重复开仓。

### 9.4 手数管理：统一 / 独立（表 3）

每个节点的实际下单手数由**该品种**在 `node.filters` 中的 `lot_mode` 决定（v0.4 起：未在 `filters` 配置该品种的节点在 9.1 候选筛选阶段即被跳过，不会进入手数计算）：


| lot_mode | 取值来源                | 说明                 |
| -------- | ------------------- | ------------------ |
| `global` | `config:lot:global` | 全局统一手数（中控台）       |
| `fixed`  | `node.filters[symbol].lot` | 该节点该品种独立手数   |
| `signal` | `signal.volume`     | 跟随信号自带手数         |


```python
def resolve_volume(node, signal_volume, global_lot, symbol):
    sf = node.filters.get(symbol)  # 按币种查找，兼容券商后缀
    mode = sf.lot_mode if sf else node_default_lot_mode  # 默认 fixed
    ...
```

- 轮询顺序同理：`node.filters[symbol].poll_order`，缺省为 0。
- 是否参与 sync/poll：读 `node.filters[symbol].follow_sync` / `follow_poll`，缺省为 `True`。

### 9.5 全员同步模式（表 7）

- 语义：一条信号让**所有**通过过滤的节点**并发同时**执行（广播）。与轮询模式的区别：同步是"所有节点都执行"，轮询是"只有轮到的 1 个节点执行"。
- 实现：分发引擎遍历过滤后的目标节点，逐一下发 `command`（进程内直接 `ws.send_json`；多实例下 `PUBLISH cmd:{node_id}`）。
- 仅 `node.filters[symbol].follow_sync=True`（缺省 True）的节点参与。

```python
async def dispatch_sync(signal):
    targets = await filter_targets(signal)        # 9.1~9.3
    for node in targets:
        cmd = build_command(node, signal)         # 含 9.4 手数
        await send_to_node(node.node_id, cmd)
```

### 9.6 轮询轮转领取模式（表 8）

- **语义（已确认）**：所有参与该品种轮询的节点**按固定顺序排成一个轮转队列**；一条信号到来后，**只被轮转队首第一个能真正成交的节点领取消费**（一个信号 = 一个节点开仓）。领取成功后，该节点被**移动到队尾**，其余节点保持原相对顺序；其他节点等待**下一条**新信号按序领取，如此**循环轮转**（round-robin 派单）。
- **默认初始顺序**按 `poll_order` 升序、其次 `created_at` 升序，顺序可在管理端调整。
- 与全员同步的区别：同步是"所有节点**并发同时**下单"；轮询轮转是"每条信号**只有轮到的 1 个节点**下单"——用于把信号/持仓**分摊**到不同账户轮流承接（分散风险、错峰建仓、规避同一时刻同向下单被风控识别）。
- 排队与推进（队列保证信号有序，轮转保证节点公平轮流）：

```
入队:   新信号 LPUSH signal:poll:pending（多条信号之间串行处理）
建任务: 取出信号，读取该品种轮转顺序 signal:poll:rotation:{symbol}，
        与“当前参与节点”对齐（剔除已失效、追加新节点到队尾）
选人领取:
        for node in 轮转顺序:
            若节点离线/被禁用            → 顺延下一个（保持其位置）
            向该节点下发 command，等待 trade_result/ACK（带超时/重试）
            被 9.2/9.3 过滤 skipped      → 顺延下一个（保持其位置）
            重试后仍失败 failed          → 顺延下一个（保持其位置）
            成功 done                    → 记为 consumer，跳出
收尾:   consumer 移动到队尾并持久化轮转顺序；signal.status = done
        （整轮无人可成交 → status = unconsumed，不下单）
```

设计要点：

- **轮转范围**：**按品种独立**维护轮转顺序（`signal:poll:rotation:{symbol}`），各品种互不影响。
- **只移动消费者**：仅**真正开仓成功**的节点移到队尾；被跳过/离线/未轮到的节点保持原位（离线节点因此会保留其轮转位置，恢复在线后优先被轮到）。
- **顺延容错**（表 8"队列错误容错处理"）：队首节点若离线、被 9.2/9.3 过滤、或失败/超时按 `POLL_MAX_RETRY` 指数退避重试后仍未成交 → **顺延给下一个候选节点**，直到有节点成交；确保"一个信号尽量被一个节点承接"。
- **逐节点实时判定**：每个节点轮到时，仍按 9.2/9.3 用该节点**最新快照**实时判定区间与持仓，保证"无持仓才开仓"等规则持续生效。
- **参与资格**：仅 `node.filters[symbol].follow_poll=True`（缺省 True）且已配置该品种、已启用的节点进入轮转队列。

> 模式切换：按品种在 `config:filters[symbol].dispatch_mode` 设置 `sync | poll`（中控台 UI）；不再提供全局 `/api/config/dispatch` 端点。

### 9.7 一致性、幂等与并发控制

多节点实盘最大的风险是**重复下单**与**竞态**。采用"全链路 at-least-once 投递 + 幂等去重"达到 effectively-once 效果：

- **信号去重**：对 webhook 原文（或 `action+symbol+volume+来源+时间窗` 指纹）计算 key，`SET dedup:{fingerprint} 1 NX EX 5` 命中即丢弃，吸收 TradingView 的重发；通过后才分配 `signal_id`。
- **指令幂等**：每条 `command` 携带 `signal_id`；节点本地保存近期已处理 `signal_id`（LRU + TTL），重连或重发时去重，**绝不因重连而重复下单**。
- **执行串行锁**：对 `(node_id, symbol)` 加分布式锁 `lock:exec:{node_id}:{symbol}`（`SET NX PX`），在锁内完成"读快照 → 持仓判定(9.3) → 下发 → 等回执"，消除"判定无持仓"与"实际成交"之间的竞态。
- **陈旧窗口处理**：开/平仓后节点**立即补报**快照（7.3）；后台在锁内对该 `(node, symbol)` 置 `pending-hold` 直到回执或超时，期间拒绝同标的再次开仓，弥补约 5s 快照间隔的空窗。
- **状态机**：信号 `pending → dispatching → done | partial | failed`；单节点结果 `done | failed | skipped`，全部持久化（见 5.2），便于追踪与对账。
- **节点重启**：节点重连后先补报快照、再继续接收指令；轮询轮转顺序 `signal:poll:rotation:{symbol}` 持久化于 Redis，后台重启后延续同一轮转进度（9.6）。

## 10. MT5 操作规则（与库代码保持一致）

> 本章逐条对齐参考库 `mt5_client.py`，节点客户端**直接复用**该模块，保证"MT5 操作规则一致"。

### 10.1 连接管理

- `mt5.initialize()` 初始化；失败记录 `mt5.last_error()`。
- 连接后读取 `account_info()`（login/server/balance/equity/margin/free_margin/leverage）。
- `is_connected()` = `connected and terminal_info() is not None`。

### 10.2 符号解析 `resolve_symbol`（与库一致）

- 处理券商后缀：TradingView 的 `XAUUSD` 可能对应 MT5 的 `XAUUSD.m` / `XAUUSD+`。
- 先大小写不敏感精确匹配；再前缀/去后缀部分匹配；多匹配时取最短名（基础符号）。

### 10.3 开仓 BUY / SELL（与库一致）

- 解析符号 → `symbol_select(symbol, True)`。
- 取价：**BUY 用 ask，SELL 用 bid**。
- 体积校验：低于 `volume_min` 或高于 `volume_max` 直接报错返回。
- 下单请求参数（与库完全一致）：

```python
request = {
    "action": mt5.TRADE_ACTION_DEAL,         # 市价成交
    "symbol": symbol,
    "volume": volume,
    "type": mt5.ORDER_TYPE_BUY,              # 或 ORDER_TYPE_SELL
    "price": price_info["ask"],              # SELL 用 bid
    "deviation": Config.DEFAULT_SLIPPAGE,    # 默认 10
    "magic": magic or Config.DEFAULT_MAGIC_NUMBER,  # 默认 20240615
    "comment": comment or "TV Signal",
    "type_filling": filling_mode,            # 自动探测：FOK→RETURN→IOC
}
# 仅当 sl/tp > 0 时才加入 request["sl"] / request["tp"]
```

- **填充模式探测**（与库一致）：读 `symbol_info.filling_mode` 优先选 FOK，否则 RETURN，否则 IOC；取不到时回退 RETURN。
- 结果判定：`retcode == TRADE_RETCODE_DONE` 为成功，返回 deal/order/volume/price。

### 10.4 平仓（与库一致）

- `close_position(ticket, volume=None)`：按持仓反向市价单平仓，`position=ticket`，BUY 持仓用 SELL 平、SELL 持仓用 BUY 平，价格相应取 bid/ask；默认平全部手数。
- `close_all_positions(symbol=None)`：遍历（可按 symbol 过滤）逐个平仓。
- `close_by_magic(magic)`：按魔术号批量平仓。
- **CLOSE 信号语义（与库 `execute_trade` 一致）**：收到 `action=CLOSE` 即对该 `symbol` 执行 `close_all_positions(symbol)`。

### 10.5 查询

- `get_positions(symbol=None)`、`get_position_by_ticket(ticket)`、`get_symbol_info(symbol)`、`get_current_price(symbol)`——字段与库一致，用于上报与平仓。

### 10.6 默认参数（与库 `config.py` 一致）


| 参数                     | 默认值      |
| ---------------------- | -------- |
| `DEFAULT_LOT`          | 0.1      |
| `DEFAULT_SLIPPAGE`     | 10       |
| `DEFAULT_MAGIC_NUMBER` | 20240615 |
| `MAX_LOT_SIZE`         | 1.0      |
| `MAX_POSITIONS`        | 5        |

> 注：参考库中 `MAX_POSITIONS` 已定义但**未在代码中启用**；本系统在风控/分发层显式使用（如需要可配合 9.3 限制单账户最大持仓数）。

### 10.7 错误码与重试策略

节点对 `order_send` 的返回 `retcode` 分类处理，**重试只在节点本地针对同一 `signal_id` 进行**（不回传后台重发，避免重复单）：

| 类别 | 典型 retcode | 处理 |
| --- | --- | --- |
| 成功 | `DONE`(10009) / `DONE_PARTIAL`(10010) | 记录 deal/order，回执 success |
| 可重试（瞬时） | `REQUOTE`(10004)、`PRICE_CHANGED`(10020)、`PRICE_OFF`(10021)、`TIMEOUT`(10012)、`TOO_MANY_REQUESTS`(10024)、`CONNECTION`(10031) | **刷新报价**后退避重试（默认 3 次：200ms / 500ms / 1s） |
| 填充模式不支持 | `INVALID_FILL`(10030) | 按 FOK→RETURN→IOC 切换 `type_filling` 重试（配合 `_get_supported_filling_mode`） |
| 业务致命（不重试） | `NO_MONEY`(10019)、`INVALID_VOLUME`(10014)、`INVALID_STOPS`(10016)、`INVALID_PRICE`(10015)、`MARKET_CLOSED`(10018)、`TRADE_DISABLED`(10017)、`INVALID`(10013) | 立即判 `failed`，回执错误并**告警**到管理端 |

- **刷新报价**：可重试类在每次重试前重新 `get_current_price`（BUY 用 ask / SELL 用 bid），必要时放宽 `deviation`。
- **结果为 None**：`order_send` 返回 `None` 时读取 `mt5.last_error()` 记录，按连接类错误处理（重连 MT5 后重试）。
- **重试边界**：超过最大次数仍失败 → `failed` + 告警；轮询模式下按 9.6 跳过该节点、继续下一个，不阻塞整体。
- **审计**：每次 `order_send` 的 `retcode/comment/deal/order/price/volume` 落 `signal_dispatch`（5.2），用于对账与定位。

```bash

# 3) 首次/升级数据库迁移
```

要点：
- **WS 有状态**：`backend` 默认单副本。需多副本时先落地第 5 章 Redis Pub/Sub 路由，再在 Nginx 对 `/ws/` 做连接级会话保持。

- **密钥管理**：密码仅放 `.env`（加入 `.gitignore`）；更高安全可改用 Docker secrets。
- **时区/编码**：容器统一 `Asia/Shanghai`，MySQL `utf8mb4`。

## 18. 设计决策确认

> 第一轮评审已逐条确认如下设计点（作为开发依据）：

1. **持仓过滤范围（表 5/6）**：✅ 两种范围都实现，做成可配置项 `config:dispatch.position_scope` 运行时切换，**默认 `symbol` 级**（同品种有持仓才禁止再开仓，与参考库一致；`account` 级为账户有任意持仓即禁止）。
2. **多区间方向过滤的配置层级（表 4）**：✅ 由人工在管理端配置，支持**全局配置**与**节点级配置**两级；**节点未单独配置时回落到全局配置**。判定价格通过取该节点实时获取品种价格。
3. **轮询轮转语义（表 8）**：✅ 参与该品种轮询的节点排成轮转队列，一条信号**只由队首第一个能成交的节点领取消费**（一个信号 = 一个节点开仓），领取后该节点移到队尾循环轮转；队首离线/被过滤/失败则**顺延**下一个节点。**默认初始顺序按 `poll_order`、其次 `created_at` 升序**，可在管理端调整；区别于全员同步的"所有节点并发同时执行"。
4. **手数回落（表 3）**：✅ 节点为 `global` 但全局手数未启用时，回落到信号自带手数 `signal.volume`，并受 `MAX_LOT_SIZE` 约束。
5. **CLOSE 与过滤**：✅ 平仓信号不参与区间/持仓/方向总开关过滤，但仍须品种已在中控台登记；对持有该品种的在线节点广播平仓。
6. **历史持久化**：✅ 实时态用 Redis；审计与历史信号落 **MySQL**，默认保留 **90 天**。
7. **后台部署形态**：✅ 初期**单实例**起步（WS 连接进程内管理）；如需高可用多实例，再按第 5 章接 Redis Pub/Sub 路由。
8. **节点与账户关系**：✅ **一个节点 = 一个 MT5 账户**（一对一）。
